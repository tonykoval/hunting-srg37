Chapter 8 · Part II Custom Backtracking

A generic SAT solver wastes most of its effort on bookkeeping a problem-specific search can skip. The last attack drops the solver and extends a fixed neighbourhood by hand, with SRG-aware propagation — fast in principle, brutal in practice.

8.1   The extension problem, restated

From Chapter 3: fix a vertex $v$ and an $(18,8)$-regular graph $H$ as its neighbourhood $N(v)$. The remaining $18$ vertices form the non-neighbourhood; every edge among the $36$ non-$v$ vertices is either forced or free, subject to:

  • every vertex has degree $18$;
  • every edge lies in exactly $8$ triangles;
  • every non-edge has exactly $9$ common neighbours.

Deciding whether a given $H$ extends to a full $\mathrm{SRG}(37,18,8,9)$ is the atomic question. If even one of the $30{,}122$ unaccounted $H$ from $\S$3.3 extends, that extension is a $\#6803$.

8.2   The backtracker

Order the undecided cells, branch on edge / non-edge, and propagate aggressively after every assignment. The two SRG-specific propagators do the real work: 

# sketch — extend a fixed neighbourhood H to a full SRG(37,18,8,9)
def extend(state):
    if not propagate(state):       # degree saturation + CN bounds
        return None                # dead end
    cell = pick_cell(state)        # fail-first: most-constrained pair
    if cell is None:
        return state if is_srg(state.A) else None
    for val in (1, 0):             # try edge, then non-edge
        child = assign(state, cell, val)
        result = extend(child)
        if result is not None:
            return result
    return None

def propagate(state):
    # 1. Degree saturation: if a vertex already has 18 neighbours,
    #    all its remaining cells are forced to 0; if it needs all
    #    remaining cells to reach 18, force them to 1.
    # 2. Common-neighbour bounds: for each decided pair (i,j) with
    #    target t in {8,9}, the count of *possible* common neighbours
    #    must stay >= t and the count of *certain* ones <= t.
    return not contradiction

The CN bound is the strong one: as cells are fixed, every pair's achievable common-neighbour interval $[\,\text{certain},\,\text{certain} + \text{possible}\,]$ tightens, and any pair whose interval excludes its target $8$ or $9$ prunes the branch immediately.

The Python prototype settles the feasibility of Python, not of the method: at $220$ nodes per second, against a search tree with hundreds of free cells, even a known-good $H$ (one we lifted from a catalog graph, so we know it extends) did not finish in $30$ seconds. The constant factor is everything. Re-encoding the same search into kissat did not help — that is the $910\text{K}$-variable wall of $\S$4.5.

8.4   The Rust port

The natural fix is a compiled backtracker with the partial assignment held in $64$-bit word bitsets, so propagation is bit-parallel: neighbourhood intersections (for common-neighbour counts) become popcount(adj[i] & adj[j]), and degree saturation is a masked word scan. This is the production path begun in the companion repository (orbit-gen v1.0.514). It lifts throughput by orders of magnitude — enough to make scanning thousands of unaccounted $H$ per CPU-day plausible — but a single full extension of a generic rigid $H$ remains genuinely hard, because the tree is enormous even with strong pruning.

Empirical finding

Purpose-built extension backtracking is the right shape of attack — SRG-aware propagation prunes far more than a generic solver — but it is constant-factor-bound. Python is hopeless at $220$ nodes/s; a Rust bitset implementation is the credible vehicle, and is the single most actionable next step (Chapter 10). With $30{,}122$ concrete candidate neighbourhoods already in hand, this is the attack most likely to either find $\#6803$ or start ruling candidates out one by one.
